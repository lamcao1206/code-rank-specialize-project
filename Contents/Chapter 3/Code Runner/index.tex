\section{Submission Evaluation}

\subsection{Submission Service}
\subsubsection{Overview}
Submission Service is a core backend component responsible for managing the lifecycle of user submissions, from initial reception to execution orchestration and result collection. It acts as an intermediary between the API Server and the execution infrastructure, ensuring that all submitted code is processed securely, efficiently, and in isolation.

When a user submits a solution through the Web Client, the request is first handled by the API Server and forwarded to the Submission Service. The service validates the submission metadata, including the programming language, problem identifier, and execution constraints. After validation, the submission is serialized and enqueued into a message queue, allowing asynchronous and scalable processing by worker services.

This queue-based architecture decouples submission handling from execution, improving system reliability under high load and preventing execution delays from affecting user-facing services.

In the deployed architecture, execution orchestration is implemented as an asynchronous pipeline across the Backend API, RabbitMQ, and Worker Service running on Google Kubernetes Engine (GKE). User code is never executed inside the API process. Instead, each submission is evaluated in a short-lived, isolated runner container created on demand as a Kubernetes Job.

The orchestration flow for each submission is as follows:

\begin{itemize}
\item The Backend API validates the submission request, stores initial submission metadata, and publishes an evaluation message to RabbitMQ.
\item The Worker Service consumes the message, resolves the testcase source, and generates a runner configuration file containing timeout, memory limit, entry function, and submission identifier.
\item The worker selects the language-specific runner image (Python, Java, or C\#) from Artifact Registry.
\item A Kubernetes ConfigMap is created to provide execution inputs (submitted source code, testcase file, and runner configuration) to the runner.
\item The worker creates a Kubernetes Job in the runner namespace; Kubernetes schedules a fresh Pod to execute the selected runner image.
\item The runner compiles/interprets the submission, executes all testcases under configured resource limits, and emits a structured JSON result.
\item The worker reads Job output from Pod logs, transforms the result into the system message format, and publishes it back to RabbitMQ for backend persistence.
\end{itemize}

This Job-based model matches the production deployment: runners are not long-running services and no fixed container pool is maintained. Every submission is executed in an ephemeral container context, which improves isolation, supports horizontal scaling through Kubernetes scheduling, and simplifies lifecycle management by terminating execution environments after completion.

\subsubsection{Security Enforcement and Attack Mitigation}

Security enforcement in the submission evaluation pipeline is achieved through a clear separation of responsibilities between system components. Docker containers used for code execution are managed exclusively by the \textbf{Submission Service}, which acts as the sole authority for container creation, allocation, and lifecycle management. User-facing components and external clients have no direct access to the container runtime.

By centralizing container control within the Submission Service, the system ensures that untrusted user code is executed only within strictly confined sandbox environments. All containers are launched with restricted privileges, isolated file systems, and predefined resource limits. This design prevents user submissions from interacting with the host operating system or with other concurrent executions.

Common attack vectors targeting CodeRank platforms are mitigated as follows:

\begin{itemize}
\item \textbf{Resource exhaustion attacks}: CPU time, memory usage, and process count are constrained at the container level, preventing infinite loops and fork bombs from degrading system stability.
\item \textbf{Unauthorized file system access}: Containers operate within isolated workspaces with read-only root file systems, ensuring that sensitive host files and judge infrastructure remain inaccessible.
\item \textbf{Privilege escalation attempts}: All container processes run as non-privileged users with restricted system capabilities, reducing the risk of kernel-level attacks.
\item \textbf{Cross-submission interference}: Containers are either destroyed or fully cleaned before reuse, preventing data leakage and persistent malicious state.
\end{itemize}

Distributed denial-of-service (DDoS) attacks are addressed at the system boundary rather than at the execution layer. The API Server applies request rate limiting, authentication checks, and input validation to regulate incoming submission traffic. By filtering excessive or malicious requests before they reach the Submission Service, the system prevents execution resources from being overwhelmed by high-volume attack patterns.

This layered security approach ensures that each component enforces protections appropriate to its role: the API Server defends against external traffic-based attacks, while the Submission Service enforces strict execution isolation for untrusted code. Together, these mechanisms provide defense-in-depth for the overall CodeRank system.

\subsection{Code Runner}
\subsubsection{Overview}
\textbf{Code Runner} Component is a core component of Submission Worker Service, responsible for executing user-submitted programs against predefined test cases. It serves as a bridge between the user’s solution and the problem’s input–output data, ensuring fair and consistent evaluation. 
For each programming language supported by the platform (C++, Java, Python), a specific implementation of the \textbf{Code Runner} will be developed to handle language-specific compilation, execution, and output parsing. This ensures that user code is run correctly and consistently, regardless of the language chosen.

\subsubsection{Class diagram}
The CodeRunner components are structured into several key classes below:

\begin{itemize}
    \item \textbf{ISerialization \& Implementations}: 
    The \texttt{ISerialization} interface provides a unified mechanism for converting raw JSON inputs and outputs into Python values. Its concrete implementations — \texttt{IntSerialization}, \texttt{FloatSerialization}, \texttt{StringSerialization}, \texttt{ListSerialization}, and \texttt{VoidSerialization} — handle serialization for specific data types, ensuring that test arguments and expected results are correctly interpreted before execution.

    \item \textbf{TestExecutor}: 
    The \texttt{TestExecutor} prepares and invokes the user’s \texttt{solve()} function for each test case. It maintains parameter serializers (\texttt{param\_traits}), return value serializers (\texttt{ret\_trait}), and parameter names. Its responsibilities include parsing the function signature, deserializing inputs, executing the function, validating outputs, and reporting mismatches with detailed context.

    \item \textbf{TimedExecutor}: 
    The \texttt{TimedExecutor} ensures safe and controlled execution of user code by applying time and memory limits. It internally uses a \texttt{Timer} to measure runtime duration and memory consumption. If a function exceeds time or memory constraints, or encounters an error, \texttt{TimedExecutor} reports the incident back to the \texttt{TestExecutor}.

    \item \textbf{Timer}: 
    The \texttt{Timer} class records performance metrics for each test case. It captures start and end timestamps, execution duration in microseconds, and memory usage. These measurements are later incorporated into the final test result records.

    \item \textbf{CodeRunner}: 
    The \texttt{CodeRunner} coordinates the overall testing process. It holds a \texttt{TestConfig} instance describing execution parameters and a \texttt{TestExecutor} instance responsible for running individual test cases. For each test case, \texttt{CodeRunner} delegates execution to the executor and collects the results.

    \item \textbf{TestConfig}: 
    The \texttt{TestConfig} class stores configuration data for the testing session, including the test data directory, test file name, timeout limit (milliseconds), and memory limit (MB). This configuration is used by both \texttt{CodeRunner} and \texttt{TimedExecutor} during execution.

    \item \textbf{TestResult \& TestResultStatus}: 
    The \texttt{TestResult} class aggregates the outcomes of all executed test cases. For each case, it records the test index, execution duration, memory usage, status code, and any raised exceptions. The \texttt{TestResultStatus} enumeration defines possible outcomes such as \texttt{PASSED}, \texttt{FAILED}, \texttt{TIMEOUT}, \texttt{STACK\_OVERFLOW}, and \texttt{RUNTIME\_ERROR}.

    \item \textbf{Solution}: 
    The \texttt{Solution} class represents the user submission and provides a single entry point, \texttt{solve(args)}, which is executed by the \texttt{TestExecutor} for each test case after deserialization.
\end{itemize}

\subsubsection{Testcase Format}
Each problem in the Online Judging System is accompanied by a test case file named \textit{problem\_slug.csv}. This file defines a structured list of input–output test pairs used by the Test Runner of the Worker Service to evaluate user submissions automatically. Moderators or superadmins upload this file when creating or updating a problem. The judging system parses the CSV, runs user code against each test case, and compares the output with the expected result.

To ensure accurate and consistent evaluation of user submissions, each input and output value in the \textit{problem\_slug.csv} file is associated with a data type. The type system defines how each field should be parsed, validated, and compared during the judging process. This mechanism helps the Test Framework interpret user program outputs correctly, whether they are numbers, strings, booleans, or lists. The following table describes the supported type tokens.

Each problem’s test case file will contain both type definitions for input, expected output and test data. The first line of the file specifies the data types for all inputs and outputs, helping the Code Runner correctly parse and validate values before running tests.

After executing all test cases for a user’s submission, the Test Runner generates a structured JSON result file summarizing the outcomes of each test. This file serves as the standardized output for the judging system and can be used by other services (such as the Result Service or API Gateway) to record, display, or analyze user performance. The JSON format ensures that results are machine-readable, language-independent, and easy to integrate into the overall online judging pipeline. The exported JSON file contains both metadata (e.g., problem name, language, execution status) and detailed per-test results (e.g., input, output, expected result, duration, and status).
