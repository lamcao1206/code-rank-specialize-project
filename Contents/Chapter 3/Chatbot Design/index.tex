\section{Chatbot Design}

\subsection{Overview}

A chatbot is an artificial intelligence software component designed to simulate conversation with human users through natural language interfaces.\cite{chatbot_definition} In CodeRank, the chatbot is designed as an AI-powered learning assistant for programming practice and technical interview preparation. Its purpose is not only to answer general questions, but also to guide users through algorithmic thinking, recommend suitable practice problems, and help users understand mistakes in failed submissions.

The chatbot is implemented as an AI agent orchestrated by n8n and powered by Gemini as the underlying large language model. Instead of directly connecting the frontend to the AI model, CodeRank routes chatbot requests through the backend and an n8n workflow. This design allows the chatbot to combine language-model reasoning with real data retrieved from CodeRank services, such as solved problems, problem metadata, and failed submission records.

The main design objectives of the chatbot are:
\begin{enumerate}
    \item Understand user requests related to algorithms, data structures, problem practice, and submission debugging.
    \item Maintain short-term conversational context across multiple messages in the same session.
    \item Use backend tools when factual system data is required, rather than generating unsupported answers.
    \item Provide responses that are educational, concise, and suitable for interview preparation.
    \item Separate AI orchestration logic from frontend and backend business logic.
\end{enumerate}

\subsection{n8n Agent Architecture}

The chatbot subsystem is designed around the main chatbot agent workflow in n8n. This workflow acts as an orchestration layer between the CodeRank backend, Gemini, short-term memory, and several tool workflows that retrieve data from backend APIs.

\textbf{Main workflow components:}
\begin{itemize}
    \item \textbf{Webhook Trigger:} Receives chatbot requests from the backend through an HTTP POST request. The request body contains the user message as \texttt{chatInput}, the conversation key as \texttt{sessionId}, and the authenticated user identifier as \texttt{userId}.
    
    \item \textbf{AI Agent Node:} Controls the chatbot behavior. It receives the user message, applies the system message, decides whether a tool call is necessary, and generates the final response.
    
    \item \textbf{Gemini Chat Model:} Provides the language understanding and reasoning capability used by the agent to explain concepts, infer user intent, and compose natural-language responses.
    
    \item \textbf{Simple Memory:} Maintains short-term conversation context using \texttt{sessionId} as the memory key. The configured context window stores the most recent 20 messages, allowing the chatbot to respond consistently within the same conversation.
    
    \item \textbf{Workflow Tools:} Provide controlled access to backend data. These tools are invoked by the agent only when the user request requires factual information from the CodeRank system.
\end{itemize}

This architecture supports a clear separation of responsibilities. The backend remains responsible for authentication, authorization, and domain APIs. n8n is responsible for agent orchestration and tool routing. Gemini is responsible for reasoning and response generation. The frontend is responsible for presenting the conversation to the user.

\subsection{Agent Tool Design}

The agent is connected to several tool workflows. Each tool workflow encapsulates one backend operation and exposes a restricted input interface to the AI agent. This design prevents the model from directly accessing arbitrary backend endpoints and ensures that tool calls remain aligned with predefined system capabilities.

The use of these tool workflows is important because several chatbot responses depend on user-specific data. For example, when a user asks for solved problems, the chatbot must call the solved-problem tool and return only the records received from the backend. Similarly, when a user asks for unsolved recommendations, the chatbot must rely on the recommendation tool because the backend is responsible for filtering out already solved problems.

\subsection{System Prompt Design}

The system prompt defines the role, capabilities, and tool-selection rules of the chatbot agent. In CodeRank, the prompt is designed to reduce hallucination by requiring the agent to call tools for factual system data and by prohibiting the generation of fake problem identifiers, URLs, solved statuses, or submission records.

\begin{tcolorbox}[
    colback=white,
    colframe=black,
    boxrule=0.8pt,
    breakable,
    sharp corners,
    listing only,
    listing options={
        basicstyle=\ttfamily\scriptsize,
        breaklines=true,
        columns=fullflexible
    }
]
\begin{verbatim}
You are Tutor Agent for coding interview practice.

Capabilities:
- Explain algorithm/data structure topics
- Recommend problems by topic/tag/difficulty
- List solved problems for the current user
- Recommend unsolved problems for the current user
- Provide hints
- Explain bugs in failed submissions
- Explain correct/optimal approaches
- Analyze testcase and complexity issues

SOLVED LIST MODE
If user asks which problems they solved, solved list, solved problems,
or Vietnamese requests like "toi da solved bai nao",
"danh sach bai da giai", "bai da lam":
1. Call `Get Solved Problems Tool`.
2. Pass current webhook userId automatically.
3. Return solvedProblems from the tool with title/name,
   difficulty/level, slug, and URL.
4. Include total if returned.
5. If solvedProblems is empty, say no solved problems were found.
6. Never invent solved problems, IDs, slugs, URLs, or solved status.

UNSOLVED RECOMMENDATION MODE
If user asks for problems they have not solved yet,
unsolved problems, remaining problems, next practice problems,
or Vietnamese requests like "bai chua solved", "bai chua lam",
"goi y bai chua giai":
1. Call `Get Unsolved Recommendations Tool`.
   Do not use `Get Problem List` for this mode.
2. Pass current webhook userId automatically.
   Infer keyword/topic and level from the message.
   Default limit to 5.
3. Use returned `recommendations` only.
   These are already filtered against `solvedProblemIds` by backend.
4. If recommendations are empty, say there are no matching unsolved
   recommendations for that filter.
   Ask whether to broaden topic/difficulty.
5. Return up to 5 recommendations with title/name,
   difficulty/level, slug, URL, and a short reason.
6. Never invent solved status, problem IDs, slugs, URLs,
   or tool results.

TOPIC MODE
If user asks about a topic, explain core concepts, patterns,
mistakes, and complexity. If they ask for practice problems,
call `Get Problem List` and return top 5 usable results.
If no usable result, generate 3 related practice problems
manually without slug or URL.

PROBLEM DEBUG MODE
If user provides a problem slug and asks for hint, explain bug,
correct approach, why wrong, or optimize solution:
1. Call `Resolve Problem Tool`.
2. If not found, ask user to clarify and stop.
3. Call `Get Failed Submission Tool`.
4. If no failed submission, inform user and stop.
5. Infer mode:
   hint/goi y -> hint;
   explain bug/why wrong/WA/TLE -> explain_bug;
   correct approach/optimal -> correct_approach;
   default -> explain_bug.
For all debug responses include problem name, problem slug,
and failed submission id.

Never hallucinate tool results.
\end{verbatim}
\end{tcolorbox}

This prompt intentionally separates the chatbot behavior into modes. Each mode defines when the agent should answer directly and when it must call a tool. As a result, the chatbot can provide flexible tutoring while still preserving factual correctness for user-specific data.

\subsection{Supported Interaction Scenarios}

The chatbot supports four main interaction scenarios that correspond to the current n8n agent implementation.

\textbf{Solved problem lookup:} When the user asks which problems they have solved, the agent calls \texttt{Get Solved Problems Tool}. The response must be based only on the backend result.

\begin{quote}
\textbf{User:} Which problems have I solved?\\
\textbf{Chatbot:} You have solved 3 problems: Two Sum (Easy), Binary Search (Easy), and Longest Substring Without Repeating Characters (Medium). Here are their slugs and URLs.
\end{quote}

\textbf{Unsolved recommendation:} When the user asks for problems they have not solved yet, the agent calls \texttt{Get Unsolved Recommendations Tool}. The backend filters recommendations against the user's solved problem list.

\begin{quote}
\textbf{User:} Recommend five medium dynamic programming problems I have not solved.\\
\textbf{Chatbot:} Here are five medium dynamic programming problems that are still unsolved in your account. I recommend starting with the first one because it focuses on the basic state-transition pattern.
\end{quote}

\textbf{Topic explanation and practice:} When the user asks about an algorithmic topic, the agent can explain the concept directly. If the user asks for practice problems, the agent calls \texttt{Get Problem List}; if no usable backend result is available, it may generate practice prompts without system-specific slugs or URLs.

\begin{quote}
\textbf{User:} Explain two pointers and give me some practice problems.\\
\textbf{Chatbot:} Two pointers is a technique that uses two indices to scan a sequence efficiently. I will retrieve related practice problems and suggest the most relevant ones for this topic.
\end{quote}

\textbf{Failed submission debugging:} When the user provides a problem slug and asks why their answer is wrong, the agent first resolves the problem, then retrieves the latest failed submission for the authenticated user. The final answer includes the problem name, slug, failed submission identifier, and debugging guidance.

\begin{quote}
\textbf{User:} Why is my submission wrong for two-sum?\\
\textbf{Chatbot:} I found your latest failed submission for Two Sum. The issue is that the current approach may reuse the same index twice. A correct solution should store previously seen values and check whether the complement already exists.
\end{quote}

\subsection{Design Benefits}

The proposed chatbot design provides several important benefits for the CodeRank system:
\begin{itemize}
    \item \textbf{Modularity:} Agent behavior, memory, model configuration, and backend tool access are separated into workflow components.
    
    \item \textbf{Maintainability:} Prompt rules and tool-routing logic can be updated in n8n without major changes to frontend or backend code.
    
    \item \textbf{Factual reliability:} User-specific information is retrieved from backend tools instead of being generated by the language model.
    
    \item \textbf{Extensibility:} New AI capabilities, such as contest assistance, personalized study plans, or retrieval-augmented generation, can be added by introducing new workflow tools.
    
    \item \textbf{Observability:} n8n provides execution-level visibility, making it easier to inspect tool calls, model responses, and workflow failures during testing.
\end{itemize}

Overall, the chatbot design enables CodeRank to provide a practical AI tutor for coding interview preparation while keeping the AI subsystem modular, observable, and aligned with the platform's existing backend services.
